# Security architecture

## Authentication model
- The API is stateless (`SessionCreationPolicy.STATELESS`).
- Authentication uses JWT bearer tokens in the `Authorization` header.
- `JwtAuthenticationFilter` runs before `UsernamePasswordAuthenticationFilter` and sets the Spring Security context when token validation succeeds.

## Token lifecycle
- Tokens are generated by `JwtTokenProvider` during `/api/v1/auth/login`.
- Token subject is the normalized email; roles are serialized in a `roles` claim.
- Secret and expiration are bound from `security.jwt.*` properties (`JwtProperties`).

## Login/register flow
- `AuthController` exposes:
  - `POST /api/v1/auth/login`
  - `POST /api/v1/auth/register`
- Registration normalizes email, hashes password (BCrypt), and assigns `ROLE_USER`.
- Duplicate email registration is detected through DB constraint `ux_users_email_lower` and mapped to domain exception.

## Authorization model
### Public routes
- `/api/v1`
- `/api/v1/auth/**`
- Swagger/OpenAPI static routes
- Webhook route `/api/v1/webhooks/**`
- Public catalog GET routes (products/categories browse/search)

### Protected routes
- All remaining routes require authentication.
- `/api/v1/admin/**` additionally requires role `ADMIN`.
- Controllers also use method-level `@PreAuthorize` for explicit module-level intent.

## Webhook-specific security note
- CSRF is disabled for `/api/v1/webhooks/**`.
- Stripe webhook authenticity is verified in `PaymentServiceImpl` with Stripe signature verification and business-level payload checks.
